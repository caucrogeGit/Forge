Une forge pour les créer toutes.

Framework web applicatif Python.

JSON canonique. SQL visible. CRUD généré. Authentification. Relations. Starters progressifs.

Forge lit mvc/entities/<entity>/<entity>.json, produit du SQL explicite, génère contrôleurs, formulaires et vues Jinja2, gère authentification, sessions et routes protégées, puis vous laisse les choix importants. Pas d’ORM magique, pas de fichiers métier écrasés.

Un CRUD MVC généré, mais jamais enfermant.

Forge génère les projections répétitives et garde visibles les choix importants : modèle JSON, SQL, routes, fichiers générés et fichiers préservés.

JSON

Source canonique JSON

Le modèle réel vit dans mvc/entities/<entity>/<entity>.json. C’est la source lisible qui pilote la génération.

SQL

SQL visible

Forge génère des fichiers *.sql et relations.sql inspectables avant application. Le schéma reste proche du réel.

DRY

Génération sûre avec --dry-run

forge build:model --dry-run montre ce qui serait écrit avant de toucher aux projections régénérables.

MVC

CRUD explicite avec forge make:crud

Contrôleur, modèle applicatif, formulaire et vues Jinja2 sont générés comme squelette modifiable, avec CSRF dans le formulaire.

OK

Diagnostic avec forge doctor

forge doctor vérifie l’environnement, les fichiers attendus, la configuration et les prérequis avant d’aller plus loin.

UP

Service d’upload sécurisé

Validation MIME, extension, taille, protection path traversal. forge upload:init prépare les dossiers de stockage.

⚓

Déploiement guidé

forge deploy:init génère la conf Nginx et le service systemd. forge deploy:check valide l’environnement de production.

Démarrer par le chemin officiel.

Créez un projet, vérifiez l’environnement, générez le modèle, appliquez le SQL puis produisez le CRUD MVC.

1
Créer le projet
$ forge new MonProjet
2
Entrer dans le projet
$ cd MonProjet
3
Activer l’environnement
$ source .venv/bin/activate
4
Diagnostiquer le projet
$ forge doctor
5
Initialiser la base
$ forge db:init
6
Créer l’entité canonique
$ forge make:entity Contact --no-input
7
Ajouter les champs métier
Après make:entity --no-input, le JSON contient seulement la clé primaire. Ajoutez les champs métier avant de générer le modèle.
$ nano mvc/entities/contact/contact.json
8
Prévisualiser la génération
$ forge build:model --dry-run
9
Écrire les projections régénérables
$ forge build:model
10
Appliquer le SQL
$ forge db:apply
11
Générer le CRUD MVC
$ forge make:crud Contact
12
Lancer l’application
$ python app.py
Note : forge new prépare le projet, l’environnement local et la configuration de développement. MariaDB doit être disponible avant forge db:apply.

Quatre starters progressifs.

Chaque starter avance d’un cran : d’abord le CRUD simple, puis la sécurité, les relations et enfin une application métier plus dense.

Niveau 1

Contacts

Apprendre le cycle complet d’un CRUD sur une seule entité : JSON, SQL, formulaire, contrôleur, vues et messages flash.

Niveau 2

Utilisateurs et authentification

Comprendre sessions, login, logout, CSRF, routes publiques et routes protégées dans une application Forge.

Niveau 3

Carnet de contacts relationnel

Lire un modèle relationnel explicite avec many_to_one, pivot many-to-many explicite et requêtes SQL visibles.

Niveau 4

Suivi pédagogique

Vitrine Forge complète : authentification, dashboard protégé, élèves, cours, observations et relations SQL visibles.

Questions fréquentes.

Forge privilégie les fichiers visibles et les décisions explicites.

Que génère make:crud ?

Un contrôleur, un modèle applicatif, un formulaire, un layout et les vues Jinja2 index, show et form. Le formulaire contient le token CSRF.

Pourquoi les routes ne sont pas injectées automatiquement ?

Parce que les routes sont une décision applicative. Forge affiche le bloc prêt à copier, puis vous choisissez où et comment l’intégrer dans mvc/routes.py.

Que signifie fichier généré / fichier préservé ?

*_base.py, *.sql et les vues CRUD générées sont régénérables. Les fichiers métier manuels, comme contact.py, sont préservés.

Pourquoi --dry-run ?

Pour vérifier l’impact d’une génération avant écriture. C’est particulièrement utile quand le modèle JSON évolue ou quand plusieurs entités changent.

Pourquoi pas d’ORM ?

Forge veut garder le SQL visible et le développeur au centre. Les modèles applicatifs peuvent contenir les requêtes dont le projet a besoin, sans couche implicite qui masque les tables, les jointures ou les contraintes.

Comment gérer des fichiers uploadés ?

Lancez forge upload:init pour créer les dossiers de stockage. Appelez save_upload(file, category=…) dans le contrôleur. La validation MIME, l’extension et la taille sont automatiques.

Comment préparer le déploiement ?

forge deploy:init génère la configuration Nginx (reverse proxy vers 127.0.0.1:8000) et l’unité systemd dans deploy/. forge deploy:check vérifie Python, les variables DB, les dossiers et les modules avant mise en production. Aucune commande ne touche au serveur.

Ce que Forge permet réellement aujourd’hui.

Forge n’est pas un framework universel. Il vise des applications lisibles, structurées et maîtrisées, avec une sécurité HTTP explicite.

Ce que Forge fait bien

  • Back-offices simples à intermédiaires
  • Applications de gestion : contacts, clients, stocks, commandes
  • CRUD MVC structuré avec formulaires générés
  • Projets où le SQL doit rester lisible
  • Applications pédagogiques et prototypes propres
  • Sécurité HTTP : CSRF, refus 413, chemins statiques durcis
  • Uploads sécurisés : validation MIME, extension, path traversal
  • Déploiement guidé : conf Nginx + systemd générées par la CLI

Ce que Forge ne cache pas

  • Pas d’ORM riche ni de migrations opaques
  • Pas d’admin automatique type Django
  • Pas d’injection silencieuse de routes
  • Pas de many-to-many magique : utilisez un pivot explicite
  • Pas de logique métier dans les JSON
  • Pas d’écrasement volontaire des fichiers manuels
